# Git Worktree 管理

刚发现错过了 Git 的又一重大功能——worktree 功能。我觉得最好的学习就是实践，这里简单学习一下这一过程。

> 相关阅读：[[现代编程语言学习笔记]]

---

## 应用场景

- 这个分支正在开发新功能，另一个分支还要临时修 bug；
- 主分支代码要跑服务，开发分支代码也要单独调试；
- 需要同时对比两个分支的文件差异，而不想来回切换；
- 某个分支上已经有未提交修改，不敢贸然切分支。

很多人遇到这种情况，第一反应是再 `git clone` 一份仓库。确实能解决问题，但代价也很明显：占空间、管理麻烦、同步远端也不够优雅。（这个我自己其实也是明显遇到过，也就是复制了多个文件夹，取了不同名字这样的安排。）

---

## 核心命令

核心命令就是在原本仓库的基础上创建新的分支：

```
git worktree add -b feature/add-blog ../obsidian_doc-add-blog master
```

---

## 与默认方法的区别

对于 Git worktree 来说，它是在当前的 master 分支直接延伸出来的，所以不会影响我们在 main 中已有的修改。

一般的应用场景是我们需要修改一个功能，同时又不想在当前的目录下修改的时候，或者有多个修改任务的时候，我们可以各自修改各自的内容，最后再合并。这样就比较符合多个 coding 工具的管理模式。

---

## 适用情况

如果有下面这些情况，worktree 明显更舒服：

- 当前分支正在改东西，不想被打断；
- 要同时维护两个、多个分支；
- 一个分支跑服务，另一个分支开发新功能；
- 要并排对比两个分支的代码或运行效果；
- 不想反复 `stash` / `checkout`。
