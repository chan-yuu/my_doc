# SSH 公钥私钥免密登录原理

SSH 免密登录的核心是非对称加密。理解这个原理，关键是搞清楚公钥和私钥各自的职责，以及它们在认证过程中如何配合。

---

## 一、非对称加密的基础

传统加密用同一个密钥既加密又解密，密钥泄露就意味着整个系统失效。非对称加密用两个不同的密钥解决这个问题：私钥用于解密数据和签名身份，绝对保密，只有自己持有；公钥用于加密数据和验证签名，可以公开给任何人。

这两个密钥数学上关联：用公钥加密的数据，只有对应的私钥能解密；用私钥签名的数据，只有对应的公钥能验证。

常见的非对称加密算法有 RSA（传统算法，密钥较长，兼容性好）、ECDSA（椭圆曲线算法，密钥短，速度快）、ED25519（现代椭圆曲线算法，推荐使用，最安全高效）。

| 密钥类型            | 作用        | 是否可以公开      |
| --------------- | --------- | ----------- |
| 私钥（Private Key） | 解密数据、签名身份 | 绝对保密，只有自己持有 |
| 公钥（Public Key）  | 加密数据、验证签名 | 可以公开给任何人    |

## 二、免密登录的完整流程

把场景设定清楚：客户端是你用来登录的设备（比如 iPhone、Termius、你的电脑）；服务器是被登录的设备（比如你的 Windows 电脑）。

### 第一步：生成密钥对

在客户端生成密钥对：

```bash
ssh-keygen -t ed25519 -C "my_comment"
```

这会生成两个文件：`id_ed25519` 是私钥，留在客户端，绝不分享；`id_ed25519.pub` 是公钥，复制到服务器。

### 第二步：分发公钥

把公钥内容添加到服务器的 `authorized_keys` 文件：

```bash
echo "公钥内容" >> ~/.ssh/authorized_keys
```

现在服务器知道：持有这个公钥对应私钥的人，是被允许登录的。

### 第三步：认证过程

当你用 SSH 连接服务器时，发生以下对话：

客户端先说"我想登录，这是我的公钥指纹"。服务器检查 `authorized_keys`，找到匹配的公钥。服务器生成一个随机挑战串，用公钥加密后发回，说"证明你是私钥持有者"。客户端用私钥解密挑战串，返回解密结果。服务器验证解密成功，身份确认，建立连接。

这个过程的关键点有三个：服务器不直接验证密码，而是用数学挑战验证你持有私钥；私钥从不传输，始终留在客户端；公钥可以公开，即使泄露也没关系，因为只有私钥持有者才能响应挑战。

### 流程图示

```
客户端                                    服务器
   |                                         |
   |  1. "我想登录，这是我的公钥指纹"  ------>  |
   |                                         |  检查 authorized_keys
   |                                         |  找到匹配的公钥
   |                                         |
   |  2. "证明你是私钥持有者"      <---------  |  生成随机挑战串
   |                                         |  用公钥加密后发回
   |                                         |
   |  3. 用私钥解密挑战串          ------>    |  解密成功
   |     返回解密结果                          |  身份确认
   |                                         |
   |  4. "认证成功，建立连接"      <---------  |
   |                                         |
```

## 三、为什么叫"免密"

"免密"不是没有密码，而是密码变成了私钥。

密码登录时，"密码"是字符串密码，记在脑子里或密码管理器里。密钥登录时，"密码"是私钥文件，存在客户端设备上。密钥登录"免密"是因为私钥文件存在本地，SSH 客户端自动读取，不需要每次手动输入。除非你给私钥设置了 passphrase，那就还需要输入一次。

| 登录方式 | "密码"是什么 | 存储位置 |
| --- | --- | --- |
| 密码登录 | 字符串密码 | 记在脑子里 / 密码管理器 |
| 密钥登录 | 私钥文件 | 存在客户端设备上 |

## 四、Windows 管理员用户的特殊情况

Windows OpenSSH 对管理员用户有特殊安全规则：管理员用户的公钥必须放在系统级目录，用户目录的 `authorized_keys` 会被忽略。

```
普通用户:  ~/.ssh/authorized_keys        （生效）
管理员用户: ~/.ssh/authorized_keys        （不生效）
管理员用户: C:\ProgramData\ssh\administrators_authorized_keys  （生效）
```

为什么这样设计？管理员权限高，如果用户目录的配置被篡改（比如恶意软件修改 `authorized_keys`），攻击者就能获得管理员权限。系统级目录权限更严格，只有 Administrators 组和 SYSTEM 能访问。

## 五、sshid.io 工具的工作原理

sshid.io 是一个便捷的密钥分发服务。

在 iPhone 上访问 sshid.io，生成密钥对。私钥保存在 iPhone 浏览器或 App 中。公钥存储在 sshid.io 服务器，分配一个 ID。在目标机器执行：`curl https://sshid.io/cyun >> authorized_keys`。公钥下载到目标机器，完成配置。

安全性分析：公钥暴露没关系，公钥本来就是公开的；私钥存储在 iPhone 本地，sshid.io 不存储私钥；风险点是 sshid.io 被攻击时可能被替换公钥，但无法获得私钥。

## 六、常见问题

### 私钥丢失怎么办？

重新生成新的密钥对，把新公钥添加到服务器，删除旧的公钥。旧的私钥无法恢复。

### 多台设备都要登录同一服务器？

每台设备生成自己的密钥对，把所有公钥都添加到服务器的 `authorized_keys`（一行一个）。

### 如何更安全？

给私钥设置 passphrase：

```bash
ssh-keygen -t ed25519 -C "comment"
# 提示时输入 passphrase（不是留空）
```

每次使用私钥需要输入 passphrase，双重保护。可以用 `ssh-agent` 缓存 passphrase，避免重复输入。

### authorized_keys 权限必须是什么？

目录 `~/.ssh/` 权限应该是 700（只有自己能读写执行）。文件 `authorized_keys` 权限应该是 600（只有自己能读写）。权限过松，SSH 会拒绝使用该文件（安全考虑）。
